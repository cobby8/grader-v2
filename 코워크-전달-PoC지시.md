# [작업 지시] grader-v2 PoC — CMYK 완전 보존 + 경량 PDF 합성 증명

## 0. 한 문장 목적

**기준 디자인 PDF(CMYK) 1개로, 여러 조각·여러 사이즈에 반복 합성한 출력 PDF를 만들되, 입력 CMYK 색상값이 한 톨도 변하지 않고(완전 보존) 동시에 파일 크기가 비대해지지 않음(경량)을 증명한다.**

이 PoC는 본 프로젝트(스티즈 전사유니폼 출력 자동화)의 출력 엔진 핵심 가설 **딱 하나**만 검증합니다. 통과하면 나머지(웹 UI·주문서·배번)는 "껍데기 씌우기"입니다.

---

## 1. 배경 (왜 이 PoC가 필요한가)

- 원래 설계는 출력 포맷을 **EPS**로 잡고 Ghostscript `eps2write`로 변환할 예정이었습니다.
- 그런데 EPS 변환에는 검증된 치명적 한계가 있습니다:
  - **투명도가 1%라도 있으면 페이지 전체를 비트맵으로 구워버림**(공식 문서 확인).
  - **ICC 기반 색은 PostScript로 표현 불가** → 색 수치가 미묘하게 바뀜.
  - 결과물의 "속 구조가 재조립"되어 원본과 달라질 수 있음.
- **사용자가 출력 포맷을 PDF로 확정**했습니다. PDF는 투명도를 지원하고 device CMYK를 그대로 통과시킬 수 있어 위 한계 대부분이 사라집니다. → **Ghostscript eps2write 단계(설계서의 `eps.py`)는 완전히 폐기**됩니다.
- 단, PDF로 가더라도 **절대 조건 2가지**는 반드시 증명되어야 합니다:
  1. **CMYK 완전 보존** — 입력 디자인의 CMYK 색상값이 출력 PDF에서 한 톨도 변하면 안 됩니다. (RGB 변환·ICC 재매핑 금지, device CMYK 그대로 통과)
  2. **파일 크기 과대 금지** — 전 사이즈 × 전 선수를 합쳐도 파일이 비대해지면 안 됩니다.

이 두 가지가 동시에 성립하는지를 작은 스크립트로 증명하는 것이 이 PoC입니다.

---

## 2. 검증 목표 (Acceptance Criteria — 이걸 만족해야 통과)

아래 3개를 **측정 가능한 수치/로그**로 증명하세요. "잘 되는 것 같다"는 불합격입니다.

### AC-1. CMYK 완전 보존 (가장 중요)

출력 PDF 안의 디자인 색이 입력 PDF의 색과 **수치적으로 동일**함을 증명해야 합니다. 아래 **두 방법을 모두** 수행해 교차 검증하세요.

**(a) pikepdf로 색공간/색값 직접 추출 비교**
- 입력 PDF의 콘텐츠 스트림에서 색 연산자(`k`/`K` = CMYK fill/stroke, `cs`/`CS` = 색공간 지정, `scn`/`SCN`)와 그 인자값을 추출한다.
- 출력 PDF에 임베드된 동일 디자인(Form XObject 내부)의 콘텐츠 스트림에서 같은 연산자·인자값을 추출한다.
- **입력의 CMYK 4값 리스트 == 출력의 CMYK 4값 리스트** 임을 비교 출력한다.
- 색공간이 `/DeviceCMYK`로 유지되는지(또는 ICCBased로 둔갑하지 않았는지) `Resources/ColorSpace`를 확인한다.

**(b) Ghostscript inkcov(잉크 커버리지)로 교차 검증**
- `gswin64c -o - -sDEVICE=inkcov 입력.pdf` 와 출력 PDF의 디자인이 들어간 영역을 비교.
- inkcov는 페이지별 C/M/Y/K 잉크 커버리지 비율을 출력하므로, **디자인을 1:1 스케일로 단독 배치한 출력 페이지의 inkcov 값이 입력 PDF의 inkcov 값과 일치**하면 색이 보존된 것이다.
- (주의: inkcov는 렌더 기반이라 스케일·클리핑이 다르면 절대값이 달라짐. 비교용으로는 "디자인 1개를 변형 없이 그대로 올린 페이지" 하나를 따로 만들어 입력과 맞대본다.)

> 통과 기준: (a)에서 CMYK 인자값 리스트가 **완전 일치**, 색공간이 `/DeviceCMYK` 유지. (b)에서 1:1 배치 페이지의 inkcov가 입력과 **소수점 오차 수준 내 일치**.

### AC-2. 파일 크기 경량 (Form XObject 재사용)

- 디자인을 **N개 조각 × M개 사이즈**(예: N=4 조각, M=6 사이즈 = 24회 배치, 또는 더 크게)에 반복 배치한 출력 PDF를 만든다.
- **핵심 기법: 디자인을 Form XObject로 PDF에 단 1회만 임베드하고, 나머지 모든 배치는 그 XObject를 `Do` 연산자로 참조**한다. (같은 그림을 복사-붙여넣기 하는 게 아니라, "그림 1장 + 위치/크기 지정만 N×M번")
- 통과 기준(택1, 둘 다 보이면 더 좋음):
  - 출력 PDF 크기 ≈ **(디자인 1개 임베드 크기) + (배치 횟수 × 작은 상수)**. 배치 수를 2배로 늘려도 파일 크기가 거의 선형으로 조금만 증가(디자인 본체가 중복 임베드되지 않음).
  - 또는: **배치 24회 출력 PDF 크기 < 디자인 PDF 크기 × 1.5** 수준.
- 로그로 다음을 출력: `입력 디자인 크기 / 배치 횟수 / 출력 PDF 크기 / 디자인이 PDF 안에 몇 번 임베드됐는지(반드시 1이어야 함)`.

> 안티패턴(불합격): 디자인을 배치할 때마다 통째로 다시 임베드해서 파일이 배치 수에 비례해 부풀어 오르는 방식.

### AC-3. 합성 정확성 (벡터 무손실 + 클리핑/스케일)

- 각 배치가 **클리핑(`W n`으로 조각 윤곽 안쪽만 보이게)** + **스케일(`cm` 행렬로 사이즈별 확대/축소)** 으로 올바르게 배치되는지.
- 출력 PDF를 열었을 때 디자인이 벡터로 유지(텍스트 선택/확대 시 깨지지 않음)되는지.
- 이번 PoC에서 클리핑 윤곽은 **단순 사각형 또는 임의 다각형**으로 충분(실제 패턴 SVG/DXF 파싱은 범위 밖).

## 3. 권장 기술 / 접근

### 중심 도구: pikepdf

- **합성은 pikepdf로만** 한다. pikepdf는 원본 디자인을 **다시 그리지 않고 통째로 얹는(Form XObject) 무손실 합성**이라 CMYK가 변형되지 않는다. (이게 이 PoC가 성립하는 이유)
- 권장 흐름:
  1. 빈 **DeviceCMYK** 출력 페이지 생성(사이즈별 1페이지, 또는 한 페이지에 여러 조각).
  2. 입력 디자인 PDF의 페이지를 **Form XObject로 변환해 출력 PDF에 1회만 등록**(`as_form_xobject` / 페이지를 XObject로 임베드).
  3. 각 배치마다 콘텐츠 스트림에 `q` → 클리핑(`W n`) → 스케일·이동(`cm`) → `/Design Do` → `Q` 를 써서 **같은 XObject를 참조만** 한다.
  4. 저장 시 불필요한 객체 정리(`pdf.save(..., compress_streams=True)` 등)로 추가 경량화하되, **이미지/색공간을 변형하는 옵션은 절대 켜지 않는다.**

### 입력이 RGB이거나 ICC 태그가 있을 때의 방침 (중요)

- 이번 PoC의 **표준 입력은 device CMYK PDF**다. 샘플도 device CMYK로 만든다.
- 만약 입력이 RGB이거나 ICCBased 색공간을 달고 있으면:
  - **PoC에서는 색 변환을 시도하지 말 것.** 변환은 색을 바꾸는 행위이므로 "완전 보존" 가설과 충돌한다.
  - 대신 **"입력이 device CMYK가 아님"을 감지해 경고 로그를 남기고, 그 파일은 보존 보장 대상에서 제외**한다. (실제 운영에서는 "디자인을 device CMYK로 내보내 달라"는 입력 SOP로 해결할 예정 — PoC 범위 밖)

### PyMuPDF는 미리보기 전용 (출력에 절대 사용 금지)

- PyMuPDF(fitz)는 **렌더할 때 화면용 RGB로 변환**한다. → 출력 PDF 생성에 쓰면 CMYK가 깨진다.
- PyMuPDF는 **사람이 눈으로 확인하는 미리보기 PNG를 뽑는 용도로만** 쓴다(선택 사항). 미리보기 PNG의 색은 참고용이며 색 검증의 근거로 삼지 않는다(색 검증은 AC-1의 pikepdf/inkcov로만).

## 4. 함정 / 주의사항 (타당성 검토에서 확인된 사실 기반)

1. **투명도 평탄화.** PDF는 투명도를 지원하지만, 합성·후처리에서 의도치 않은 투명도(`ca`/`CA`<1, `SMask`, `BM`이 `Normal`이 아님)가 섞이면 일부 RIP/뷰어에서 문제가 생길 수 있다. PoC 샘플은 **투명도 없는 평탄(flatten) 디자인**으로 만들 것. 입력에 투명도가 있으면 스캔해 로그로 알릴 것.
2. **이미지 재압축 금지.** 디자인 안에 비트맵 이미지가 있을 때 저장 옵션이 이미지를 재인코딩하면 **색공간이 변형(CMYK→RGB 등)** 될 수 있다. pikepdf 저장 시 이미지 재압축/재샘플 옵션을 켜지 말 것. (스트림 압축 = 무손실 deflate는 OK, 색을 건드리지 않음)
3. **색공간 둔갑 방지.** 합성 과정에서 `/DeviceCMYK`가 `/ICCBased`나 RGB로 바뀌지 않는지 출력 PDF의 `Resources`를 반드시 검증(AC-1-a).
4. **폰트는 PoC 범위 밖이지만**, 텍스트가 들어간 디자인을 샘플로 쓸 경우 폰트 미임베드로 깨지지 않게 **아웃라인화되었거나 임베드된 PDF**를 쓸 것.
5. **inkcov 비교 함정.** inkcov는 스케일·클리핑·페이지 크기에 영향을 받는다. 색 보존 비교용으로는 **변형 없는 1:1 단독 배치 페이지**를 따로 만들어 입력과 맞대볼 것(AC-1-b 참조).
6. **PDF/X 호환은 이번엔 깊게 보지 말 것.** 다만 device CMYK + 투명도 없는 평탄 디자인이면 추후 PDF/X-1a 적합화가 쉬워진다는 점만 인지.

## 5. 입출력 샘플 정의 (코워크가 테스트할 수 있도록)

### 입력 (코워크가 준비)
- **샘플 CMYK 디자인 PDF 1개** (`sample_design_cmyk.pdf`):
  - **device CMYK** 색공간.
  - **투명도 없음**(평탄).
  - 색 검증이 가능하도록 **CMYK 값을 알 수 있는 단색 도형 여러 개**를 포함(예: C100, M100, Y100, K100, 그리고 C50/M20/Y0/K10 같은 혼합색 1~2개). 각 도형 옆에 색값을 텍스트로 적어두면 사람이 보기에도 좋음(텍스트는 아웃라인화).
  - 크기는 실제 유니폼 조각을 흉내 낸 적당한 사이즈(예: A4 또는 임의 mm).
- (선택) 배치할 **조각 윤곽**: 이번엔 코드 안에 사각형/다각형 좌표를 하드코딩해도 됨.

### 출력 (코워크가 생성)
1. `output_composed.pdf` — 디자인을 **N개 조각 × M개 사이즈**(예: 4×6=24회)에 클리핑+스케일로 합성한 PDF. 디자인은 내부에 **1회만 임베드**.
2. `output_1to1.pdf` — 색 검증 전용. 디자인을 **변형 없이 1:1로 한 번** 올린 PDF(AC-1-b inkcov 비교용).
3. **검증 로그**(콘솔 출력 또는 `poc_report.txt`)에 아래를 모두 포함:
   - `[AC-1a] 입력 CMYK 값 리스트` vs `출력 CMYK 값 리스트` → **일치 여부 PASS/FAIL**
   - `[AC-1a] 출력 색공간 = DeviceCMYK 유지 여부` → PASS/FAIL
   - `[AC-1b] 입력 inkcov` vs `output_1to1 inkcov` → 수치 + 일치 여부
   - `[AC-2] 입력 디자인 크기 / 배치 횟수 / output_composed.pdf 크기 / 디자인 임베드 횟수(=1이어야 함)` → PASS/FAIL
   - `[AC-2] 배치 횟수 2배로 늘렸을 때 파일 크기 증가폭`(선형·소량 증가 확인용, 선택)
   - `[AC-3] 출력 PDF 벡터 유지 / 클리핑·스케일 적용 확인`

### 산출물 정리
- 위 입력/출력/로그/사용한 스크립트를 한 폴더에 담아 전달. 스크립트 상단에 **실행 방법 1~2줄**(필요 패키지: `pikepdf`, Ghostscript 설치 경로) 적어둘 것.

## 6. PoC 범위 선긋기 (이번엔 하지 않는 것)

이번 PoC는 **"CMYK 완전 보존 + 경량 합성" 한 가지만** 증명합니다. 아래는 전부 **범위 밖**:

- ❌ 웹 UI / FastAPI 서버
- ❌ 주문서(엑셀) 파싱
- ❌ 배번·이름 텍스트 렌더(fontTools)
- ❌ 패턴 관리(SVG/DXF 실파싱) — 조각 윤곽은 하드코딩 좌표로 충분
- ❌ 원단 수축 보정, 별색(PMS), ICC 변환
- ❌ EPS / Ghostscript eps2write (폐기됨 — 쓰지 말 것)

조각 윤곽·사이즈 스케일은 **테스트가 가능한 최소한의 가짜 값**으로 충분합니다. 진짜 패턴/주문 연동은 PoC 통과 후 본 개발에서 합니다.

## 7. 이 PoC가 통과하면 무엇이 보장되는가 (비개발자용 쉬운 비유)

- **CMYK 보존이 증명되면**: 디자이너가 정한 "이 빨강은 정확히 이 빨강"이라는 약속이, 자동 생성된 모든 사이즈·모든 선수 파일에서 **한 치도 안 바뀌고 그대로** 간다는 뜻입니다. 공장 인쇄 색이 디자인과 어긋날 걱정이 사라집니다.
- **경량이 증명되면**: 디자인이라는 "원본 도장" 하나만 파일에 넣고, 나머지는 "여기·여기·여기 찍어라"는 **위치 메모만** 붙이는 방식이라, 전 사이즈·전 선수를 다 합쳐도 파일이 무겁지 않다는 뜻입니다. 다운로드·전달이 빠르고 가볍습니다.
- 한마디로, 이 PoC가 통과하면 **"색 안 변하고, 파일 안 무거운" 출력 엔진의 심장이 실제로 뛴다는 것**이 증명됩니다. 그 다음부터는 그 심장에 버튼과 화면만 붙이면 됩니다.
