| 프로퍼티 | 레이어 | 내용 |
|---------|--------|------|

| **/MC3** | **패턴선** | **재단선(실재함, 너치 포함)** |

> ⚠️ 이전에 "패턴선(재단선) 레이어가 비었다"고 본 판단은 **오판**이었다. 디자인에 실재한다.

**확정된 3대 근본 원인:**

1. **앞/뒤 svg_index 반전.** 현재 preset 은 `front→svg_index 0`, `back→svg_index 1`.
   실제로는 **정반대**여야 한다 — SVG 조각을 넥 형태로 보면 idx0(우측·넥깊이≈189)=라운드넥=**뒤판**,
   idx1(좌측·넥깊이≈409)=V넥=**앞판** (L.svg, parse_svg 가 높이 desc 정렬한 기준). 이게 합성 좌우 뒤집힘 원인.

2. **design_region_pt y하한 오류.** 현재 front/back 모두 y하한 `2606`. 실제 본체 콘텐츠는 **y2938**부터.
   빈공간 ~335pt 가 영역에 끼어 cover 해도 디자인이 덜 채워진다(흰여백).

3. **채움 방식이 contain.** 조각 안쪽으로 '맞춤(min)'이라 가장자리에 여백. **cover(꽉채움)** 여야 한다.
   (시접까지 디자인이 모두 차도록 설계된 원단이므로 흰여백은 결함.)

**완성본 대조**(`연세대_V넥_완성본_본체포함_XL.ai`): 좌=YONSEI 팀명+가슴로고=앞판, 우=이름+큰번호+1885워터마크=뒤판.

## 2. 핵심 작업 — 조각 자동매핑기 (engine/reference.py 확장)

`reference.py` 에 디자인 빈템플릿 + 패턴 SVG 를 받아 **조각별 design_region 과 svg_index 를 자동 결정**하는
함수를 추가한다. preset 의 `pieces[].design_region_pt` / `svg_index` 를 수동 입력 대신 이걸로 생성/검증.

**2-1. 디자인에서 조각 영역 추출** (`extract_design_pieces(design_pdf) -> list[bbox]`)
- 페이지 콘텐츠 스트림을 파싱, OCG 프로퍼티명→레이어명 매핑(`page.Resources.Properties` + `Root.OCProperties.OCGs`).
- BDC `/OC <name>` … EMC 로 현재 레이어 추적. **패턴선 레이어**(이름에 "패턴선"/"재단선" 포함) 안의
  닫힌 서브패스(m…f/S/n 단위)별 bbox 를 CTM 적용해 PDF native 좌표로 수집.
- 면적 큰 상위 N개(=조각 수)만 채택. 결과: 디자인 조각 bbox 목록.
- 패턴선 레이어가 없거나 비면 **몸판 레이어**로 폴백(동일 로직), 그것도 없으면 경고+기존 preset 값 사용.

**2-2. SVG 조각 형태 측정** (parse_svg 결과 활용)
- 각 SVG 조각의 bbox, 그리고 **넥 깊이**(상단 가로중앙 부근에서 어깨최상단 y 와 넥최하점 y 의 차) 계산.
  V넥=깊이 큼, 라운드=작음. 밴드=높이가 현저히 작음(전체 높이의 ~10% 이하).

**2-3. 디자인 조각 ↔ SVG 조각 매칭** (`match_pieces`)
- 밴드: 양쪽 모두 '높이 최소' 조각끼리.
- 앞/뒤(큰 두 조각): **좌우 위치 + 넥 깊이**로 매칭.
  - 디자인 좌측 영역(작은 x)=앞판(팀명쪽), 우측=뒤판. SVG 는 넥깊이 큰 쪽=앞판(V넥), 작은 쪽=뒤판.
  - 둘이 일치(좌측 디자인↔V넥 SVG, 우측 디자인↔라운드 SVG)하면 확정. 불일치 시 경고 로그.
- 산출: 조각별 `{id, design_region_pt, svg_index}`.

**2-4. CLI 명령**: `python -m engine.cli build-preset --design <빈템플릿> --svgdir <패턴폴더> --base XL`
→ pieces(design_region_pt·svg_index) 자동 채운 preset 조각 섹션 출력/갱신. (number/name area 는 기존 reference 추출 유지.)

## 3. 합성 수정 — cover 채움 (engine/job.py)

`_job_piece_transform`(line ~141) 의 contain(min) → **cover(max) + 중앙정렬**:
```python
s = max(pw / dw, ph / dh) * max(shrink_x, shrink_y)   # contain(min) → cover(max)
ox = px0 + (pw - s * dw) / 2 - s * dx0                  # 중앙정렬
oy = py0 + (ph - s * dh) / 2 - s * dy0
```
- 조각 윤곽(SVG 폴리곤) clip 은 그대로 유지 → cover 로 넘친 디자인은 조각 밖으로 안 나간다.
- grade.py `_piece_transform` 도 동일 정책으로 맞출지 확인(현재 job 경로만 쓰면 job 만 수정).

## 4. 재단선 — SVG 조각 윤곽을 빨강 stroke 로 (확실한 방식)

디자인 패턴선을 추출·보존하는 기존 시도(이슈4)는 불완전했다. **재단선 = 조각 윤곽 자체**이고
SVG 폴리곤에 너치가 이미 포함되므로, **각 조각의 SVG 폴리곤을 그대로 빨강 stroke 로 그리는 방식**이 확실.
- clip '밖'(extra_ops, 시트좌표)에서 `q / K(0 0.96 0.95 0) / w(preset 두께) / <m,l…> h / S / Q`.
- 색은 `preset.cutline.color_cmyk`([0,0.96,0.95,0]), 폴리곤 점은 `poly.points`(너치 포함).
- 검증: 출력에 빨강 재단선이 3조각 외곽 모두 + 너치 표식까지 보일 것.

> ⚠️ **재단선이 두 줄로 겹치는 문제 — 반드시 처리.** 디자인 자체에 **패턴선 레이어(/MC3)**가 들어 있어,
> 그대로 디자인을 깔면 디자인의 패턴선 + 우리가 그린 SVG 재단선이 **두 줄**로 보인다. 정밀 분석으로 확인됨.
> → **디자인을 조각에 채울 때 패턴선 OCG 를 비표시(OFF)** 로 만들 것. (pikepdf: 해당 OCG 의 /View·/Print 상태를
> OFF 로, 또는 콘텐츠에서 `/OC <패턴선> BDC … EMC` 구간을 합성 대상에서 제외.) 재단선은 §4 의 SVG 폴리곤으로만 1회 그린다.

## 4-1. 블리드(bleed) — 안쪽 흰틈 제거 (반드시 처리)

디자인은 base_size(XL) 1개뿐인데 출력 조각은 사이즈별(L 등)이라, cover 만으로는 디자인 외곽과 조각 윤곽이
가장자리에서 미세하게 어긋나 **조각 안쪽에 흰틈**이 보인다(정밀 분석으로 확인). 형태(곡선)는 거의 일치하므로
**작은 블리드로 해결**된다 — cover 배율에 **약 1.02~1.05 배(기본 1.03)** 를 곱해 디자인을 재단선보다 살짝 크게
넣고 조각 폴리곤으로 clip 한다. clip 이 항상 디자인(색) 위를 지나가 흰틈이 사라진다.
- `s = max(pw/dw, ph/dh) * BLEED`  (BLEED 는 preset 설정값, 기본 1.03). 중앙정렬 오프셋은 §3 식 그대로.
- 검증: 조각 안쪽 어디에도 흰틈 없음. (cowork 검증 이미지 `교정검증2_L.png`)
