> 선행: `의뢰서-디자인-UIUX.md`(1차) + 1차 납품물(`web/`). 이 문서는 **1차 이후 발견된 공백 3건**만 다룬다.

---

## 0. 전제 — 1차 위에 얹는다

1차 납품(`web/` : 토큰 + 화면 3개 + states + flow)은 품질이 좋고 그대로 쓴다. **다시 만들지 말 것.**
이번 작업은 기존 **디자인 토큰(`web/tokens/`)과 컴포넌트(`web/screens/app.css`)를 재사용**해 화면을 확장/추가하는 것. 같은 제약 유지: 순수 HTML/CSS/vanilla JS, 빌드체인 없음, CDN 2개(Pretendard·Material Symbols)만, 한글 UI, 데스크톱 우선.

> 특히 1차 패턴관리 화면에 이미 만든 **드래그-그리기/이동/리사이즈 영역 에디터**(배번·이름 영역)는 ①에서 그대로 재사용한다.

## 1. ① 디자인↔조각 매핑 UI  (최우선 · 가장 큰 기능적 공백)

### 배경
패턴 1종의 "레시피"인 `preset.json`에는 **`design.content_box`**(기준 디자인에서 실제 아트워크가 차지하는 영역)와 **조각별 `design_region_pt`**(디자인의 어느 영역이 앞판/뒤판/소매에 얹히는가)가 들어간다. 지금은 이 좌표를 사람이 **손으로 측정해 JSON에 입력**해야 한다 — 비개발자에게는 사실상 패턴 등록 불가 지점.

### 요구
패턴 등록 화면(`patterns.html` register 모드)에 **"디자인 영역 매핑" 단계**를 추가:
1. 기준 디자인(렌더된 이미지)을 캔버스에 표시.
2. 조각 목록(앞판·뒤판·소매…)에서 조각을 고른 뒤, **디자인 위에 드래그로 그 조각이 쓸 영역(사각형)을 지정**. → 1차의 영역 에디터 재사용(조각마다 색 구분).
3. (선택) `content_box`는 전 조각 영역의 합집합으로 자동 제안 + 수동 조정.
4. 모든 조각이 매핑됐는지 검증(미매핑 조각 경고). 진행 차단 규칙 동일.

### 통합 지점 / 데이터
- 기존 register 모드는 ⓐ 사이즈별 파일 업로드 → ⓑ **(신규) 디자인 영역 매핑** → ⓒ 배번/이름 영역 지정(기존) 순서로 확장.
- 산출 데이터(서버가 `preset.json`에 저장):
  ```
  design.content_box : [x0,y0,x1,y1]        (디자인 페이지 pt 좌표)
  pieces[].design_region_pt : [x0,y0,x1,y1] (조각별)
  ```
  → **좌표는 디자인 페이지 pt 기준.** 캔버스는 디자인을 축소 표시하므로, 드래그 좌표를 페이지 pt로 환산해 출력(디자인 페이지 크기는 서버가 제공: `{design:{page_size_pt:[w,h]}}`).

### UX 포인트
- "조각을 고르고 디자인 위에 영역을 그리세요" — 1차의 안내 톤 유지.
- 조각↔영역을 색으로 1:1 매칭, 지정 완료 조각엔 체크. 겹침/누락 시 경고.
- y축 방향(디자인은 좌상단 원점 이미지 ↔ 출력은 좌하단 원점 pt) 혼란이 없도록 라벨/가이드 제공.

## 2. ② 검수 상세(줌) 모달  (A-4 배번·이름 가치와 직결)

### 배경
현재 검수 그리드(work.html panel-4)는 작은 썸네일 + PASS/확인 배지뿐. 곧 배번·이름이 **자동 렌더**되면 "이름·배번이 제자리에 맞게 박혔는지"가 핵심 확인 대상이 된다. 한글 이름·긴 이름·특수 배번을 **확대해 검수**할 화면이 필요.

### 요구
검수 그리드의 결과 1건을 클릭 → **상세 모달**:
- **큰 미리보기**(전체 유니폼 렌더) + 배번·이름 영역 **확대/포커스** 보기.
- **검증 항목 리스트**: 각 `check`의 이름·통과여부·상세. PASS는 차분히, FAIL은 무엇이/어떻게 고치는지(예: 투명도 → 평탄화 안내).
- 단일 파일 다운로드 + 이전/다음 결과 이동(키보드 ←/→).
- 닫기/배경클릭. 1차 모달 컴포넌트 재사용.

### 통합 지점 / 데이터
- work.html panel-4 + states.html에 specimen 추가.
- 데이터(이미 계약에 있음):
  ```
  RESULTS[] : {size, name, number, url, checks:[{name, ok, detail}]}
  ```
  `checks`는 출력 엔진 verify 결과(무손실 보존·단일 임베드·CMYK 유지·투명도·합성 정확) 그대로. `url`은 미리보기 PNG.

### UX 포인트
- 모달 제목에 `사이즈 · 이름 · 배번`. 전달 가능/확인 필요를 상단에 명확히.
- 배번·이름 확대는 "맞게 들어갔나"를 한눈에 — 영역 경계 오버레이(preset의 number/name 영역) 옵션 고려.

## 3. ③ 대량 주문 UX  (실사용 필수)

### 배경
작업기록 mock에 48·64명 주문이 있다. 주문표·검수 그리드는 6~8행 기준이라 수십~수백 선수에서 무너진다.

### 요구
**주문표(work.html panel-2)**:
- 검색(이름/배번), 필터(**확인 필요만 보기**), 정렬(사이즈/이름).
- 페이지네이션 **또는** 가상 스크롤 + 고정 헤더(sticky). 행 수 요약("64행 · 확인 필요 3").
- 인라인 수정·행 추가/삭제는 유지. 대량에서도 빠른 입력 동선.

**검수 그리드(work.html panel-4)**:
- 상단 필터 탭(전체 / 전달 가능 / 확인 필요) + 카운트.
- 검색(이름/배번), 페이지네이션/스크롤, 결과 수 요약.
- "확인 필요만" 빠른 필터 → 실패분만 모아 처리하는 동선(부분 실패 허용 규칙과 연결).

### 통합 지점 / 데이터
- 기존 `dtable`·`tabs`·`badge`·`tile` 컴포넌트 재사용. ORDER/RESULTS 배열이 커지는 것만 가정.
- 빈/필터-결과-없음 상태는 1차 states 재사용.

## 4. 산출물 형식 (1차와 동일)
1. 확장된/추가된 **화면 HTML/CSS**(기존 `web/` 구조·토큰·app.css 재사용, 바로 결합).
2. 신규 **상태 specimen**(②③ 관련)을 states에 추가.
3. 변경/추가 **컴포넌트 명세**(특히 ① 매핑 에디터, ② 상세 모달).
4. 짧은 **변경 요약**(어떤 파일을 어떻게 확장했는지).

## 5. 범위 밖
백엔드 로직·실제 결합(CLI 담당)·모바일 우선·인증·④ states 추가건(이번 제외 — 이름 오버플로우/글리프 누락/연결 끊김은 차기).
