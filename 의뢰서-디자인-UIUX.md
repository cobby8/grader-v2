# 의뢰서 — UI/UX 디자인 (grader-v2)

> 수신: 디자인 클로드  |  발신: 코워크(허브)  |  작성일: 2026-06-15
> 협업 구조: **코워크가 조율**, **CLI가 구현**, **당신(디자인 클로드)이 UI/UX를 설계**한다.
> 당신의 산출물은 CLI 구현자가 정적 HTML/CSS로 받아 FastAPI에 붙인다. 즉 **바로 구현 가능한 풀 비주얼**이 목표.

---

## 0. 제품 한 줄 설명

스티즈 전사유니폼 출력 자동화 웹앱. 직원이 브라우저에서 **주문 1건을 패턴 선택 → 디자인/주문서 업로드 → [생성] → 미리보기 확인 → ZIP 다운로드**로 끝내면, 사이즈별·선수별 공장 출력용 파일이 자동 생성된다. 기존엔 Illustrator에서 사이즈마다 수작업하던 것을 없앤다.

## 1. 사용자와 맥락

- **주 사용자 = 사무실 직원(비개발자).** 매 주문마다 반복하던 그레이딩/배번/이름 입력을 줄이는 게 목적. 빠르고, 실수 없고, "지금 뭘 해야 하는지"가 분명해야 한다.
- **관리자(가끔) = 패턴 등록자.** 새 패턴을 최초 1회만 등록(사이즈별 파일 업로드 + 배번/이름 위치 지정).
- **환경**: 사내 PC 브라우저. 데스크톱 우선(모바일 불필요). 한글 UI.
- **톤**: 신뢰감 있는 실무 도구. 화려함보다 **명료함·안정감**. 인쇄/제조 현장 도구다운 차분함.

## 2. 핵심 시나리오 (이 흐름이 화면 설계의 척추)

```
직원이 접속 →
 1. 패턴 선택      (등록된 패턴 프리셋 중)
 2. 디자인 업로드   (기준 사이즈 1개, PDF호환 AI/PDF)
     └ 투명도/색공간 자동 점검 → 문제 시 경고(진행 차단)
 3. 주문서 업로드   (엑셀 → 사이즈·수량·이름·배번 자동 추출)
     └ 추출된 표를 화면에서 확인/수정
 4. [생성] 클릭     → 진행률 표시 (사이즈×선수별 생성)
 5. 미리보기 검수   → 각 결과 PNG + 검증 결과(공장 전달 가능 여부) 확인
 6. ZIP 다운로드   → 공장 전달
```

## 3. 설계할 화면 3개

### 화면 1 — 작업 화면(메인). **가장 중요.**
위 시나리오 1~6을 한 흐름으로. 단계형(스텝) 또는 세로 흐름 등 UX 판단은 당신 몫.
요구 요소:
- **패턴 선택**: 썸네일 카드 목록에서 선택.
- **디자인 업로드**: 드롭존. 업로드 후 상태:
  - 정상: 파일명 + 미리보기.
  - **투명도 경고**(중요 엣지케이스): "이 디자인은 투명도가 있어 출력 시 래스터화됩니다. 원본에서 평탄화 후 재업로드하세요." → 진행 버튼 비활성. 친절하지만 단호하게.
  - 색공간/형식 오류 안내(예: PostScript AI 거부).
- **주문서 업로드 → 추출 표**: 자동 추출된 `사이즈 / 수량 / 이름 / 배번` 행들을 **편집 가능한 표**로. 인라인 수정, 행 추가/삭제. 추출 실패 셀 강조.
- **[생성] + 진행률**: `완료 N / 전체 M`, 사이즈별 진행. 취소 가능.
- **미리보기 그리드**: 결과(사이즈×선수)별 PNG 카드. 각 카드에 **검증 결과 배지** — 전부 PASS면 "공장 전달 가능", FAIL 있으면 어떤 항목인지(예: 투명도) 표시.
- **[ZIP 다운로드]**.

### 화면 2 — 패턴 관리
- **패턴 목록**: 카드/리스트(썸네일, 이름, 지원 사이즈, 수정/삭제).
- **신규 등록 흐름**: 패턴명 → 사이즈별 파일 업로드(XS~5XL, SVG/DXF) → **배번/이름 위치 지정**.
  - 위치 지정 UX(핵심): 패턴 미리보기 위에 **드래그로 영역을 그려** 배번 영역·이름 영역을 지정. 영역마다 정렬/크기/색 옵션. 이게 이 화면의 난이도 높은 부분 — 직관적 인터랙션 설계 필요.

### 화면 3 — 작업 기록
- 지난 주문 목록(날짜, 주문명, 패턴, 상태).
- 행별 **재다운로드**, **같은 설정으로 재생성**.

## 4. UX 요구사항

- **비개발자 친화**: 전문용어 최소화, 각 단계 "지금 할 일"이 분명. 다음 단계는 이전이 끝나야 열리는 점진적 노출 고려.
- **오류·경고가 친절하면서 명확**: 무엇이 문제인지 + 어떻게 해결하는지(특히 투명도 평탄화 안내).
- **진행 피드백**: 업로드/생성은 시간이 걸린다 → 로딩·진행률·예상 상태를 분명히.
- **검증 결과의 가시성**: "이 결과를 공장에 보내도 되는가"가 한눈에. 이 제품의 신뢰는 여기서 나온다.
- **상태/엣지케이스 전부 디자인**: 빈 상태(패턴 없음/작업 없음), 로딩, 부분 실패, 투명도 경고, 추출 실패, 다운로드 준비 중.

## 5. 비주얼 요구사항 (풀 비주얼까지)

- **컬러 시스템**: 1차/보조/배경/표면 + 의미색(성공=전달가능, 경고=투명도, 오류, 정보). 인쇄 도구다운 차분한 팔레트.
- **타이포그래피**: 한글 본문 가독성 우선(예: Pretendard 등 시스템 친화 웹폰트). 위계(제목/본문/캡션/숫자).
- **컴포넌트**: 버튼(주/보조/위험), 드롭존, 카드, 편집 표, 배지/칩(검증 상태), 진행률, 모달(경고), 토스트, 빈 상태, 탭/스텝.
- **레이아웃**: 데스크톱 기준 그리드, 여백 체계, 반응형은 데스크톱 폭 위주(모바일 비필수).
- **아이콘**: 가벼운 라인 아이콘 세트 권장.

## 6. 구현 제약 (디자인이 이 안에서 구현 가능해야 함)

- **정적 HTML + CSS + vanilla JS로 구현된다. 프레임워크·빌드체인 없음**(React/Tailwind 컴파일 금지). → CSS는 순수 CSS 또는 CDN 한 장 수준에서 동작 가능하게.
- 상태는 서버(FastAPI)에 있음. 브라우저 `localStorage`에 업무 데이터 저장 가정하지 말 것(필터/정렬 기억 정도는 가능).
- 화면 3개 규모라 과한 디자인 시스템보다 **실용적이고 일관된** 토큰/컴포넌트가 적합.

## 7. 백엔드 API 계약 (당신의 화면이 맞물릴 데이터 형태)

CLI 구현자와 공유하는 계약이다. 화면은 이 데이터 모양을 가정해 설계하면 결합이 매끄럽다.

| 화면 요소 | 데이터(요지) |
|-----------|-------------|
| 패턴 목록 | `[{id, name, sizes:["XS"…], thumb}]` |
| 디자인 업로드 결과 | `{ok, transparency:[…], colorspace_ok}` ← 경고 표시용 |
| 주문 추출 표 | `{rows:[{size, qty, name, number}]}` (편집 가능) |
| 생성 진행률 | `{done, total, state}` |
| 미리보기 그리드 | `[{size, name, url, checks:[{name, ok, detail}]}]` ← 카드+검증 배지 |
| 작업 기록 | `[{id, date, name, pattern, status}]` |

> `checks`는 출력 엔진의 자동 검증 결과(무손실 보존·단일 임베드·CMYK 유지·투명도·합성 정확). 화면에서 "전달 가능/불가"로 번역해 보여주면 된다.

## 8. 산출물 형식 (이렇게 주면 CLI가 바로 받음)

1. **화면별 목업** — 가능하면 정적 **HTML/CSS**(바로 구현 결합), 또는 고해상 이미지 + 명세.
2. **상태/엣지케이스 화면** — 4번에 적은 케이스들 포함.
3. **디자인 토큰** — 색/타이포/간격/반경/그림자 변수 정의.
4. **컴포넌트 명세** — 각 컴포넌트의 상태(기본/호버/비활성/오류)와 용법.
5. **UX 플로우 다이어그램** — 3개 화면 간 이동 + 작업화면 단계 흐름.
6. **한글 카피 톤 가이드**(짧게) — 경고/안내 문구 톤.
