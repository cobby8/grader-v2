# grader-v2 설계서 — 스티즈 전사유니폼 출력 자동화 (완전 신규 설계)

> 작성일: 2026-06-10 | 상태: 설계 확정 대기
> 원칙: **기존 grader 코드는 한 줄도 가져오지 않는다.** 검증된 지식(규칙·교훈)만 이식한다.

---

## 1. 목적

주문이 들어오면 **기준 사이즈 디자인 1개로 전 사이즈·전 선수의 공장 출력용 벡터 CMYK EPS를 자동 생성**한다.
Illustrator는 출력 과정에서 완전히 배제한다 (디자인 제작 도구로만 남음).

## 2. 용도 — 주문 1건의 흐름 (사용자 시나리오)

```
[주문 접수]
  직원이 브라우저에서 grader-v2 접속
    1. 패턴 선택      : 등록된 패턴 프리셋에서 선택 (예: U넥 양면 스탠다드)
    2. 디자인 업로드   : 기준 사이즈 디자인 (PDF 호환 AI/PDF) 1개
    3. 주문서 업로드   : 엑셀 — 사이즈·수량·이름·배번 자동 추출 → 표로 확인/수정
    4. [생성] 클릭     : 사이즈×선수별 EPS 일괄 생성 (사이즈당 수 초)
    5. 미리보기 검수   : 화면에서 각 파일 PNG 미리보기 확인
    6. 다운로드/전달   : ZIP 다운로드 → 공장 전달
[완료]
```

패턴 프리셋 등록(사이즈별 SVG/DXF 업로드 + 배번/이름 위치 지정)은 최초 1회만 하는 관리 작업.

## 3. 범위

**포함**: 그레이딩 + 벡터 CMYK EPS 출력(코어), 주문서 자동 인식, 배번/이름 자동 입력, 패턴 프리셋 관리(SVG/DXF), 미리보기 검수.
**제외** (기존 grader에 있었으나 의도적으로 뺌): G드라이브 자동 동기화, AI→SVG 일괄 변환, 자동 업데이트 시스템, Supabase 인증(외부 확장 시점에 추가). — 기능이 아니라 복잡도의 근원이었던 것들.

## 4. 형태 결정: 중앙 서버 웹앱

| 기준 | A. 데스크톱 앱 (기존 방식) | B. PC별 로컬 웹앱 | **C. 중앙 서버 웹앱 (채택)** |
|------|--------------------------|------------------|------------------------|
| 직원 PC 요구사항 | 설치 + Python + 업데이트 | Python + Ghostscript 설치 | **브라우저만** |
| 업데이트 배포 | 자동업데이트 시스템 구축 필요 (기존 15h+) | PC마다 재설치 | **서버 1곳 코드 교체** |
| 빌드 체인 | Rust+Node+Python 3중 | Python | **Python 단일, 빌드 0** |
| 외부 확장 | 불가능 | 불가능 | **같은 코드로 클라우드+로그인만 추가** |
| 바이브 코딩 수정 난이도 | 높음 (기존 실패 원인) | 중간 | **낮음** |

운영 형태: 사무실 PC 1대(또는 클라우드 VM)에서 서버 실행 → 직원은 `http://서버주소:8000` 접속. Python과 Ghostscript는 **그 한 대에만** 설치.

## 5. 기술 스택 (전부 Python 생태계, 빌드 단계 0)

| 역할 | 선택 | 이유 |
|------|------|------|
| 웹 서버 | FastAPI + Uvicorn | 표준적, 자료 풍부, 바이브 코딩 친화 |
| 화면 | 순수 HTML + vanilla JS (정적 파일) | React/빌드체인 배제. 페이지 3개 수준이라 충분 |
| PDF 합성 | pikepdf | PoC 검증 완료 (클리핑·CMYK·스케일) |
| EPS 변환 | Ghostscript eps2write | PoC 검증 완료 (벡터·CMYK 보존) |
| 미리보기 | PyMuPDF (PNG 렌더) | 검증 완료 |
| 패턴 파싱 | xml(SVG) + ezdxf(DXF) | PoC 검증 완료 |
| 배번/이름 글자 | fontTools (글리프→벡터 경로) | 폰트 임베드 불필요, 공장 호환 안전 |
| 주문서 | openpyxl | 기존 order_parser 방식 검증됨 |
| 데이터 저장 | 폴더 + JSON 파일 | DB 없음. 패턴 수십 개·주문 건별 폴더 수준이라 충분, 백업=폴더 복사 |

## 6. 데이터 모델 (개념 3개만)

```
data/
├── patterns/<패턴이름>/
│   ├── preset.json          # 사이즈 목록, 조각 정의, 배번/이름 영역 좌표(조각 기준 상대 비율)
│   └── XS.svg … 5XL.svg     # 사이즈별 패턴 윤곽선 (SVG 또는 DXF)
├── fonts/                   # 배번/이름용 폰트 (ttf/otf)
└── jobs/<날짜_주문명>/
    ├── job.json             # 선택한 패턴, 주문 내역(사이즈·이름·배번), 상태
    ├── design.pdf           # 업로드된 기준 디자인
    └── output/              # 생성된 EPS + 미리보기 PNG
```

- **Pattern(패턴 프리셋)**: 사이즈별 조각 윤곽 + 배번/이름이 들어갈 위치·크기·색 규칙.
- **Job(작업)**: 주문 1건 = 패턴 + 디자인 + 주문 행들. 모든 산출물이 한 폴더에.
- 사용자/권한 개념 없음(사내 단계). 외부 확장 시 이 위에 로그인만 얹음.

## 7. 처리 파이프라인 (PoC로 검증된 경로)

```
engine/ (웹과 완전 분리된 순수 Python 모듈 — 단독 테스트 가능)
1. pattern.py   : SVG/DXF → 조각별 윤곽 점 목록 (큰패턴/작은패턴/절단선 분류)
2. order.py     : 엑셀 → [{size, qty, name, number}] (셀 전체 스캔 방식)
3. compose.py   : pikepdf — 빈 CMYK 페이지에 조각별 [클리핑 W n → 스케일 cm → 디자인 인라인]
4. text.py      : fontTools — 이름/배번 글리프를 벡터 경로로 변환해 지정 좌표에 CMYK fill
5. eps.py       : Ghostscript eps2write 호출 + 자동 검증(벡터인지/CMYK인지/BoundingBox)
6. preview.py   : PyMuPDF로 검수용 PNG 렌더
```

**투명도 안전장치**: 디자인 업로드 시 `ca/CA<1`, `SMask`, `BM≠Normal` 스캔 → 발견 시 "이 디자인은 투명도가 있어 EPS가 래스터화됩니다. 원본에서 평탄화 후 재업로드하세요" 경고 후 진행 여부 선택.

## 8. 화면 (3개)

1. **작업 화면(메인)**: 패턴 선택 → 디자인 업로드 → 주문서 업로드 → 추출된 표 확인/수정 → [생성] → 진행률 → 미리보기 그리드 → [ZIP 다운로드]
2. **패턴 관리**: 패턴 목록, 신규 등록(사이즈별 파일 업로드), 배번/이름 위치 지정(미리보기 위에 드래그로 영역 지정)
3. **작업 기록**: 지난 주문 목록, 재다운로드, 같은 설정으로 재생성

## 9. 기존 grader에서 이식하는 "지식" (코드 아님)

| 지식 | 출처 | 적용 |
|------|------|------|
| AI 파일 헤더 분기 (`%PDF-` vs `%!PS-Adobe`) | lessons 2026-04-21 | 디자인 업로드 검증. PostScript AI는 거부+안내 |
| 조각 분류는 x위치보다 크기(높이) 기준이 안정적 | lessons 2026-04-21 | pattern.py 분류 규칙 |
| 사이즈 키워드 긴 것부터 매칭 (5XL→XL 순) | order_parser.py | order.py |
| 주문서 셀 전체 스캔(가로형/세로형/표형) | order_parser.py | order.py |
| 단순 비례 스케일 + 클리핑이 업계 허용 오차 내 | decisions 2026-04-08 | compose.py |
| 멱등성 테스트 + 실행 전 시뮬레이션 | lessons 2026-04-21 | engine 전 모듈 테스트 원칙 |
| 투명도→EPS 래스터화, 원본 1회 평탄화로 해결 | PoC 2026-06-10 | 업로드 검증 |
| openpyxl Custom Properties 버그 워크어라운드 | order_parser.py | order.py |

## 10. 로드맵

| Phase | 내용 | 완료 기준 |
|-------|------|----------|
| 0 | 프로젝트 뼈대 + engine 모듈 골격 + 테스트 파일 복사 | `python -m engine.compose` 단독 실행으로 PoC 재현 |
| 1 | engine 완성: 조각별 합성(전 사이즈) + EPS + 검증 | 기존 일러 출력물과 시각 비교 통과, 공장 시험 인쇄 1회 |
| 2 | 웹 UI: 작업 화면 (패턴 선택~ZIP 다운로드) | 직원이 브라우저로 주문 1건 처리 |
| 3 | 주문서 인식 + 배번/이름 렌더 | 실제 주문서 3종으로 검증 |
| 4 | 패턴 관리 화면 + DXF 입력 | 신규 패턴을 UI만으로 등록 |
| 5 | (외부 확장 시) 클라우드 이전 + 로그인 | 같은 코드, 인프라만 변경 |

각 Phase는 독립적으로 동작·확인 가능해야 하며, engine은 웹 없이 커맨드라인으로 항상 테스트 가능하게 유지한다.

## 11. 리스크

| 리스크 | 대응 |
|--------|------|
| 일러 출력물과 미세 차이 | Phase 1에서 동일 입력 병행 생산 → 공장 시험 인쇄로 승인 후 전환 |
| 서버 PC 꺼짐/장애 | 시작 프로그램 등록 + data/ 폴더 주기 백업(클라우드 이전 시 자동 해소) |
| 패턴 양식별 조각 분류 차이 (V넥 등) | preset.json에 조각 정의를 명시(자동 분류 의존 최소화) |
| 폰트 라이선스 | 배번/이름 폰트는 임베드 아닌 아웃라인화라 부담 적음. 상용 폰트는 라이선스 확인 |
