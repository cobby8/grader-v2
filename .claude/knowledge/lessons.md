### [2026-07-09] degrade(실패해도 진행) 부가기능은 정상경로 실 e2e를 반드시 1회 검증
- **분류**: lesson
- **발견자**: pm
- **내용**: [Phase B] Storage 백업이 실서버에서 **100% 조용히 실패**하고 있었는데, 단위테스트(39/39)·리뷰(치명0)·모킹 e2e를 전부 통과했다. 원인은 Supabase Storage가 오브젝트 키에 한글(비ASCII)을 거부(400)하는 실제 서비스 제약이라, 실 네트워크 호출 없이는 드러나지 않았다. 게다가 **degrade(실패해도 등록은 성공) 설계가 실패를 화면·로그에서 삼켜** 정상으로 보였다. 교훈: ① "실패해도 진행"하는 부가기능일수록 **정상경로가 성공하는지 실 서비스 e2e를 최소 1회** 확인해야 한다(degrade는 예외를 숨기는 것이지 정상경로 실패를 숨기라는 게 아니다). ② 외부 저장소 키에 사용자 유래 문자열(한글 패턴명)을 그대로 쓰지 말고 ASCII-safe(hex 등)로 인코딩. ③ 모킹 테스트는 "우리 코드 로직"만 검증하지 외부 서비스의 입력 검증 규칙은 못 잡는다. **실서버 브라우저 e2e(등록→버킷 확인→재배포→복원)로 최종 확정**해 백업·복원 라운드트립을 눈으로 검증함.
- **참조횟수**: 0

### [2026-06-15] 코워크(외부 환경) 산출물 인수 체크리스트
- **분류**: lesson
- **발견자**: pm
- **내용**: 코워크가 다른 환경(다른 Python 버전·다른 OS 인코딩)에서 만든 코드를 이 PC로 가져오면 "코워크 환경에선 됐는데 여기선 안 되는" 문제가 반복된다. 인수 시 표준 절차:
  1. **의존성 재현**: requirements.txt 유무 확인 → 없으면 import 전수조사로 생성 후 설치. (이번엔 pikepdf/PyMuPDF 미설치 + 명세 없음)
  2. **인코딩 검증**: 한글 Windows(cp949)에서 한 번 실행해 UnicodeEncodeError 등 콘솔 인코딩 문제 확인. (→ errors.md)
  3. **즉시 git 백업**: 인수 직후 첫 커밋으로 이력 시작(이전엔 git 미초기화 상태였음).
  4. **자체 검증 재실행**: 코워크 노트의 "샌드박스 동기화 지연" 같은 주의사항은 이 PC에서 selftest 1회 재실행으로 직접 확인.
- **참조횟수**: 0
