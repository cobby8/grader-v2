# 계획서 — 완성본 기준 → 빈 템플릿 선수별 주입

> 작성일: 2026-06-18 | 상태: 설계 확정 대기
> 목표: 디자이너가 주는 **완성본 + 빈 템플릿** 2개로, 빈 템플릿에 선수별 배번·이름을
> 자동 주입해 전사 출력용 PDF 한 벌을 깨끗하게 생성한다.

---

## 1. 배경 — 왜 이 방식인가

처음엔 "완성본(김경원 20번)에서 기존 번호를 지우고 새 번호를 넣는" 방향을 검토했다.
하지만 분석 결과:
- 이름은 살아있는 텍스트라 제거가 쉬우나,
- **번호는 윤곽선 벡터(그림에 구워짐)** 라 지우려면 정교한 콘텐츠 수술이 필요하고 디자인마다
  위험이 있다(PoC로 가능은 확인했으나 신뢰성 부담).

그래서 **사용자 확정 방식**으로 전환:
> 디자이너가 **완성본**과 **빈 템플릿(번호·이름 없는 상태)** 두 개를 제공한다.
> 시스템은 **완성본을 '기준(정답지)'** 으로 삼아 번호·이름이 들어갈 위치·크기·색을 자동 추출하고,
> **빈 템플릿(이미 깨끗한 캔버스)** 에 선수별로 주입한다.

**장점**
- 깨지기 쉬운 도형 제거가 전혀 필요 없다 → 항상 100% 깨끗.
- 번호/이름 위치 좌표를 사람이 손으로 입력할 필요 없다 → 완성본에서 자동 도출.
- 디자이너 부담은 "디자인당 빈 템플릿 1개 추가 저장"뿐.

---

## 2. 적용 범위

- **앞 번호 + 뒤 번호 + 뒤 이름** 모두 선수별 주입. (앞 번호도 변경 — 사용자 확정)
- 1차 대상: 농구 유니폼 상의. 양식/사이즈는 preset으로 확장.

---

## 3. 데이터 흐름 (완성본·템플릿 → 선수별 출력)

```
[완성본.ai]  [빈 템플릿.ai]
     │            │
     ▼            │
 (A) 기준 추출    │     완성본 - 템플릿 차이 = 번호/이름 요소
  ├ 뒤 이름: live text → Tm 위치 / 크기 / 색            │
  ├ 뒤 번호: 완성본에만 있는 윤곽선 채움 → bbox/색/높이   │
  └ 앞 번호: 〃                                          │
     │                                                  │
     ▼                                                  ▼
 (B) preset 의 area 자동 채움          (C) 빈 템플릿을 base 디자인으로
  front_number_area / back_number_area / back_name_area
  (design 좌표 rel_bbox + 색 + 정렬)
     │
     ▼
 (D) 빈 템플릿 → flatten(투명도) → job:
      선수마다 text.py 로 번호/이름 벡터 렌더 → 정위치 주입 → compose → verify
     │
     ▼
 선수별 PDF (8/8 verify PASS 목표)
```

### 핵심 아이디어 — "차이가 곧 위치"
완성본과 빈 템플릿의 **차이(diff)** 가 정확히 번호·이름이다. 이 차이를 분석하면 위치·크기·색을
**자동으로** 알 수 있어 디자인마다 좌표를 수기 입력할 필요가 없다.

---

## 4. 설계 결정 사항 / 확인 필요

| # | 항목 | 제안/현황 |
|---|------|----------|
| D1 | 번호 렌더 폰트 | 완성본 번호는 아웃라인이라 폰트명 추출 불가. preset 에 번호 폰트 지정(data/fonts). 시각적으로 맞는 폰트 1개 확보 필요(디자이너에게 폰트명 문의). |
| D2 | 이름 폰트 | 완성본 이름은 live text(H2hdrM 류). 동일/유사 한글 폰트로 렌더(Pretendard-Bold 등). |
| D3 | 번호 색 | 완성본에서 추출(흰색 0,0,0,0 등). preset 에 저장. |
| D4 | 앞/뒤 번호 동일성 | 보통 같은 번호. 한 선수 = 앞·뒤 같은 번호 주입(기본). |
| D5 | 템플릿 투명도 | 엠블럼 등 반투명 있으면 flatten 선적용(이미 구현됨). |

## 5. 디자인·패턴 제공 방법 (제안)

### (#3) 디자인 제공 폴더 구조 — 제안
```
data/designs/<디자인명>/
   reference.ai     ← 완성본(샘플 번호·이름 포함)
   template.ai      ← 빈 템플릿(번호·이름 없음)
   preset.json      ← (시스템이 생성/보완: area·색·폰트·패턴 매핑)
```
- 예: `data/designs/연세대_레플리카_V넥/reference.ai` + `template.ai`
- 디자이너는 두 .ai 만 넣으면 됨. 나머지(preset)는 `engine reference` 명령이 초안 생성.
- 여러 디자인은 폴더만 추가하면 됨.

### (#4) 패턴 위치
- 전 사이즈 패턴 .ai: `G:\공유 드라이브\디자인\2026 커스텀용 패턴\0. 농구유니폼 확정 정리본`
- 작업하려면 이 폴더를 세션에 **연결(폴더 선택)** 필요.
- 패턴 .ai(사이즈별) → SVG 변환 후 preset 의 sizes 에 매핑(A-2 전 사이즈와 연결).

## 6. 로드맵 (단계별, CLI 우선)

| Phase | 내용 | 산출물 | 선행 |
|-------|------|--------|------|
| A | **기준 추출기** `engine/reference.py`: 완성본↔템플릿 비교 → 번호/이름 위치·크기·색 추출 | `engine reference` CLI + 추출 리포트 JSON | 완성본+템플릿 1쌍 |
| B | **preset 스키마 확장**: front_number_area / back_number_area / back_name_area (+색·폰트·정렬) | preset.json 스키마 | A |
| C | **job 통합**: base=빈 템플릿, flatten 선적용, 앞·뒤 번호 + 이름 주입 | `engine job` 확장 | A,B |
| D | **V넥 preset + 패턴**: G드라이브 패턴 .ai→SVG, 전 사이즈 매핑 | V넥 preset | 폴더 연결 |
| E | **전체 검증**: 실데이터로 clean 출력 + 8/8 verify + 시각 확인 | 검증 리포트 | C,D |

각 Phase 는 CLI 한 줄로 단독 실행·검증 가능하게 만든다(기존 엔진 원칙).

---

## 7. 필요한 입력 (사용자 제공)

1. **빈 템플릿** — 이번 연세대 V넥 디자인의 번호·이름 없는 버전 1개 (Phase A 테스트용).
2. **번호 폰트명** — 완성본 번호에 쓰인 폰트(또는 가장 가까운 폰트 파일). (D1)
3. **패턴 폴더 연결** — `G:\...\0. 농구유니폼 확정 정리본` 세션 연결. (Phase D)
4. (이후) 여러 디자인 — 위 §5 폴더 구조로 추가.
